# 设计：Android 核心购物体验

Status: Implemented

## 背景

BuyWise Android 已经具备商品浏览、搜索、AI 导购、商品详情、悬浮对比篮、购买记录、待反馈提示和账号能力，但当前体验仍带有 demo 与工作台痕迹：

- 首次体验容易被登录、模式选择或过重的导购工作台打断。
- 商品流、AI 导购、详情页和对比页之间的职责边界还不够清晰。
- 部分文案会制造交易平台错觉，例如把外部购买记录表达成购物车动作。
- 推荐理由、风险点、证据来源和对比决策需要在主路径中更稳定地出现。
- 现有悬浮卡片质感偏展示型，应收敛到更清晰的购物决策卡。

本设计目标是让第一版 Android 核心 shopper journey 在体验上逼近产品级：用户可以自由浏览商品，也可以随时进入 AI 导购；BuyWise 帮用户缩短决策，而不伪装成交易平台。

## 方案

### 核心路径

第一版体验改造聚焦以下路径：

1. 用户打开 App，进入商品首页。
2. 用户可以搜索、按场景浏览商品，也可以输入预算、用途和偏好后点 `开始导购`。
3. 首页 AI 入口跳转到导购页，导购页自动生成结果。
4. 导购结果先展示商品候选，再补齐 AI 理由。
5. 用户查看首推、备选、风险点和证据来源。
6. 用户进入商品详情，查看决策摘要、证据条和购买前注意。
7. 用户将商品加入对比，使用悬浮对比篮承接多商品决策。
8. 对比页先给 AI 决策摘要，再给规格表格。
9. 用户通过 `查看商品页` 到外部商家页面完成购买。
10. 返回 BuyWise 后，用轻提示承接 `记录已购买`。
11. 购买后不强制即时评价，反馈到期后在首页轻任务卡和“我的”入口承接。

### 首页

首页保留自由浏览，不改成纯 AI 导购页。首页职责是发现商品、搜索商品、进入导购和承接待办。

首页首屏结构：

- 顶部搜索框：`搜索商品、品牌、功能`。
- 轻量 AI 导购输入：`说说你的预算、用途和偏好`，主按钮为 `开始导购`。
- 场景筛选：如 `宿舍`、`通勤`、`办公`、`厨房`、`旅行`。
- 单列商品决策流。
- 反馈到期时，在商品流上方展示轻任务卡。

首页交互规则：

- 搜索入口和 AI 导购入口不合并。搜索服务“已知目标”，AI 导购服务“需要决策”。
- 首页 AI 输入为空时不跳转，提示用户先描述需求。
- 首页 AI 输入有内容时跳转导购页，并自动开始生成结果。
- 手机首页使用单列决策流，不使用双列瀑布流。
- 商品卡非按钮区域点击进入详情。
- `加入对比` 是常驻轻按钮，加入后同步悬浮对比篮。

首页商品卡使用 BuyWise 决策卡，而不是普通电商商品卡。每张卡默认展示：

- 商品图，比例适中，不压过判断信息。
- 商品名、价格、评分、库存。
- 2 个核心标签。
- 1 条购买前注意或风险点。
- `加入对比` 轻按钮。

### 导购页

导购页职责是解释需求、处理一次必要追问、展示首推与备选、支持继续追问。

导购入口规则：

- 首页输入需求后进入导购页时自动生成，不要求二次点击。
- 导购页内的主按钮统一为 `开始导购`。
- 不在结果前强调“生成推荐”和“进入对话导购”两个模式。
- 独立聊天能力保留，但不作为主路径的前置选择。

需求不完整时：

- 只追问一次。
- 追问以内联卡片完成，不跳转聊天页。
- 追问卡只问一个关键问题，例如预算、用途或品类。
- 追问卡提供 2 到 4 个快捷选项，并保留 `直接推荐`。

生成与展示顺序：

- 不等待完整 AI 文案再展示商品。
- 系统先展示状态，例如 `正在理解需求`、`正在匹配商品`、`正在生成理由`。
- 商品匹配完成后，先出现首推和备选商品卡。
- AI 理由、风险说明和适合人群随后流式补齐。
- AI 文案失败时，商品结果仍然可用，并显示可重试解释的状态。

结果结构：

- 顶部展示可修改的需求理解条，例如 `预算 300 内`、`宿舍使用`、`低噪音`、`无线`。
- 需求理解条可触发 `修改需求`，修改后重新生成。
- 首先展示 1 个强首推，再展示 2 到 3 个备选。
- 不用 `92% 匹配` 这类百分比，避免虚假精确感。
- 使用 `匹配`、`冲突`、`需核实` 标签表达判断依据。

首推商品卡默认展示：

- 商品名、价格、评分、库存。
- 为什么首推，1 到 2 条。
- 购买前注意，至少 1 条关键风险。
- 证据标签，如 `真实商品页`、`价格快照`、`评价摘要`、`已购反馈`。
- 操作：`查看详情`、`加入对比`、结果后可 `追问这个商品`。

证据规则：

- 证据不足的商品不能作为首推。
- 首推至少需要真实商品页可打开、价格或库存快照、商品描述或评价摘要。
- 证据不足商品可作为备选，但必须标注 `证据较少，建议核实`。
- 如果没有可靠首推，不硬凑最佳推荐。页面改为低置信候选，并解释缺少的证据。

继续追问：

- 结果出来后保留 `继续追问`。
- 默认在结果页内用底部轻量输入框完成，不跳离商品上下文。
- 追问答案以内联回复卡展示，并引用当前首推和备选商品。
- 独立聊天页只作为深度多轮讨论入口，不作为第一阶段主路径。

### 商品详情

商品详情页职责是帮助用户判断某个商品是否值得继续比较，而不是模拟交易。

首屏优先展示决策摘要：

- 商品名、价格、评分、库存。
- 一句话结论：适合谁、不适合谁。
- 1 到 2 条推荐理由。
- 1 条风险点。
- 商品图适中展示，不压过判断信息。
- 默认展示紧凑证据条。

详情页主行动：

- 主行动：`加入对比`。
- 次行动：`记录已购买`。
- 若商品有真实 `product_url`，提供 `查看商品页`。
- 若没有 `product_url`，`查看商品页` 降级为 `查看详情` 或隐藏。
- 禁止使用 `加入购物车` 表达 BuyWise 内部购买记录。

证据条：

- 默认显示紧凑标签，如 `商品页可打开`、`价格快照`、`评价摘要`、`已购反馈`。
- 支持轻量展开。
- 展开内容显示来源类型、更新时间、真实商品链接可用性和反馈证据等级。
- 不暴露 RAG score、embedding source、internal id 等技术字段。
- 证据不足时明确提示：`证据较少，建议查看商家页面确认`。

### 对比页

对比页职责是缩短决策，而不是只堆规格。

页面结构：

- 顶部先展示 AI 决策摘要。
- 摘要使用明确条件表达，例如：`如果你更在意低噪音，选 A；如果你更在意价格，选 B`。
- 中部展示关键差异 3 条，例如价格、噪音、售后、便携性。
- 下方展示完整规格、评价和风险矩阵。
- 每个商品保留 `查看详情` 和 `移出对比`。

对比动作：

- 用户加入对比后不自动跳转。
- 继续停留在当前商品或结果页。
- 悬浮对比篮显示已选数量，例如 `已选 1/2`。
- 对比篮提供 `继续看备选` 和 `开始对比`。
- 选满 2 个以上后，`开始对比` 成为强按钮。

最终 CTA：

- 主行动：`查看商品页`。
- 次行动：`记录已购买`。
- 辅助行动：`查看详情`。
- `查看商品页` 使用系统浏览器或 Chrome Custom Tab，不做内置 WebView。
- 用户返回 BuyWise 后底部轻提示：`如果你已经下单，可以记录已购买`。

### 登录与游客边界

登录只保护个人数据动作，不阻断浏览和决策路径。

游客可用：

- 浏览商品。
- 搜索商品。
- 开始导购。
- 查看推荐。
- 查看商品详情。
- 加入对比。
- 查看商品页。

需要登录：

- 保存导购偏好。
- 记录已购买。
- 提交体验反馈。
- 查看历史购买和反馈。

触发登录时：

- 第一版复用“我的”页和现有账号入口，不强制登录阻断首页、搜索、导购、详情和对比。
- 购买和反馈能力不可用时，在原页面展示能力提示。
- 后续如需无跳转登录，再单独实现底部登录面板和登录后续行动恢复。
- 如果 closed beta 构建配置了评审 token，可允许游客体验受保护动作，但 UI 必须明确这是评审体验。

### 购买与反馈闭环

BuyWise 不接真实交易，不模拟购物车，不处理支付。

查看商品页：

- 购买发生在真实商家页面。
- 使用系统浏览器或 Chrome Custom Tab。
- 返回 BuyWise 后，用底部轻提示承接购买记录。

记录已购买：

- 第一阶段低摩擦一键记录。
- 可选填写购买渠道、实际价格和备注。
- 可选上传订单截图，不作为必填。
- 后端用 evidence level 表达证据强弱，前端不强迫用户提交证明。

购买后反馈：

- 记录购买后不强制立刻写完整评价。
- 可选入口为 `写购买原因`，不是即时评价。
- 到反馈窗口后再提示：`这件商品用下来怎么样？`
- 反馈表单重点问满意点、踩坑点、是否推荐和实际使用场景。
- 反馈到期后用首页轻任务卡承接，并在“我的”里保留 `待反馈商品`。
- 不使用弹窗或强红点打断浏览。

### 视觉与组件方向

Android 体验应向“清晰决策卡”收敛：

- 减少展示型大阴影、玻璃感和过度圆角。
- 商品名、价格、评分、库存、风险点和证据标签优先。
- 卡片用清晰边框、分区、状态标签和稳定间距表达层级。
- 标准控件遵循 Material 语义，不为风格重造搜索、输入、登录和筛选控件。
- 所有交互组件覆盖默认、按下、禁用、加载、错误状态。
- 加载状态优先使用骨架或分段状态，避免只有居中转圈。

### 不做范围

第一版不做：

- admin-web 体验改造。
- 多页新手引导。
- 内置 WebView。
- 真实交易、购物车、支付、退款或物流。
- 强制订单截图或订单号。
- 复杂溯源页。
- 结果前强制进入聊天模式。
- 手机双列商品瀑布流。

## 影响

### Android 页面

- `MainActivity`：首页仍为默认主入口；登录改为按需面板后需调整路由与触发逻辑。
- `HomeScreen` / 首页相关 panel：新增轻量 AI 导购入口、场景筛选、单列决策流、反馈任务卡。
- `GuideScreen`：支持首页带入 query 自动生成、内联追问、需求理解条、首推与备选结构、结果内继续追问。
- `GuideChatScreen`：保留深度对话入口，但不作为主路径前置模式。
- `ProductDetailScreen`：首屏改为决策摘要优先，主行动改为 `加入对比`，补充证据条和真实商品页入口。
- `CompareScreen`：先展示 AI 决策摘要，再展示规格表格；最终 CTA 以 `查看商品页` 为主。
- `AccountScreen` / `LandingScreen`：登录入口降级为主动入口或按需底部面板，不阻断游客核心路径。
- `FeedbackPromptCard`：反馈到期任务卡承接用后反馈。

### Android 数据与 ViewModel

- `BuyWiseViewModel`：支持首页 AI query 带入导购页并自动生成。
- `HomeViewModel`：支持场景筛选和反馈任务卡状态。
- `GuideViewModel`：支持需求理解条、一次追问、低置信推荐状态、内联追问回复。
- `ProductDetailViewModel`：支持 `查看商品页` 触发后的返回提示和一键记录已购买。
- `CompareViewModel`：支持决策摘要优先展示和最终 CTA。
- `BuyWiseApiClient` / repository 层：需要更明确的错误模型、登录触发信号、可选商品外链字段使用。

### 后端与 API

第一版优先复用现有 API。若现有响应字段不足，后续实现可小步补充：

- 商品证据摘要字段。
- 商品 `product_url` 可用性和更新时间。
- 导购需求理解结构。
- 推荐候选的 `match`、`conflict`、`needs_verification` 标签。
- 首推置信状态或低置信原因。

任何新增 API 字段必须同步 `docs/reference/api.md` 和 Android 解析测试。

### 文档

- 实现时同步 `docs/architecture/android-boundaries.md`。
- 若新增 API 字段，同步 `docs/reference/api.md`。
- 若调整登录体验边界，同步 `docs/design/user-auth.md` 或引用本设计。

## 验证

本设计的第一版主路径打磨已实现，范围是复用现有 API 和 Android MVVM 结构，聚焦首页、导购、商品卡、详情、对比篮、对比页和购买后反馈文案。按需底部登录面板、复杂溯源页、新证据 API 字段和真实交易能力不在本次实现范围。

实现验收至少覆盖：

- App 首屏保留首页商品浏览，并展示搜索、轻量 AI 导购入口、场景筛选和单列商品决策流。
- 首页 AI 输入有内容时跳转导购页并自动生成。
- 导购页需求不完整时只追问一次，并提供 `直接推荐`。
- 导购结果先展示商品，再补齐 AI 理由。
- 首推卡展示风险点和证据标签。
- 证据不足商品不能作为首推；无可靠首推时展示低置信候选。
- 商品卡和详情页不使用 `加入购物车` 表达内部购买记录。
- 商品详情主行动为 `加入对比`，并提供真实商品页入口。
- 加入对比不自动跳转，对比篮承接已选数量和 `开始对比`。
- 对比页先展示决策摘要，再展示规格矩阵。
- `查看商品页` 使用外部浏览器或 Custom Tab。
- 返回 App 后用底部轻提示承接 `记录已购买`。
- 游客可浏览、搜索、导购、详情和对比；保存偏好、记录购买和提交反馈触发按需登录。
- 不出现多页新手引导或强制登录阻断主路径。

建议验证：

- Android Compose 编译通过。
- `flutter analyze` 不适用；Android 使用 Gradle lint 或 assemble。
- 手动检查小屏设备上卡片文本不溢出，按钮不换行异常。
- 手动检查弱网、AI 文案失败、无商品、证据不足、未登录、反馈到期等状态。
- 后端或文档变更后运行 `python scripts/validate_docs.py`。
